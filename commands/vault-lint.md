볼트 자가 치유 파이프라인을 수행하라.

## 0. 전제

- 이 명령은 루트에 `00-meta/vault-config.json`이 있는 볼트에서만 동작한다.
- 검증 엔진이 "볼트 아님"을 출력하면(exit 0), 여기가 볼트가 아니라는 사실을 한 줄로 정중히 안내하고 즉시 종료하라. 다른 작업을 하지 마라.
- **절대 금지: 사용자 확인 없는 노트 삭제.** 치유는 보완·교정·이동(아카이브)까지만이며, 이동조차 관리성 이슈에서는 사용자 승인이 선행돼야 한다.

## 1. 검증 엔진 실행

```
python "${CLAUDE_PLUGIN_ROOT}/skills/agentic-vault/scripts/vault_healthcheck.py"
```

- 볼트 경로는 스크립트가 `CLAUDE_PROJECT_DIR`(없으면 cwd)에서 자동으로 읽는다. 다른 볼트를 검사하려면 `--vault <경로>`를 붙인다.
- exit 0 + 리포트 = 치명 이슈 없음(관리성만 있을 수 있음), exit 1 = 치명 이슈 존재 또는 설정 오류.
- 설정 오류(config 파싱 실패, 검사 대상 0개 등)가 출력되면 치유를 시작하지 말고 원인을 사용자에게 보고하라.

## 2. 리포트 해석

`00-meta/vault-config.json`의 `health_report` 경로(기본 `00-meta/health-report.md`)를 읽어라. 각 섹션 제목에 치명/관리성이 표기되어 있다. 섹션 0(설정 경고)이 있으면 먼저 사용자에게 보여줘라 — 설정 오류는 노트가 아니라 config를 고쳐야 한다.

## 3. 치명 이슈 — 즉시 치유 (섹션 1, 2, 3, 4, 9)

시스템(YAML 파싱·Dataview·로그 grep 계약)을 깨뜨리는 위반이므로 사용자 확인 없이 바로 고친다. 단, **내용을 지어내지 마라** — 불확실하면 보수적 기본값을 쓴다.

- **프런트매터 누락 / 필수 키 누락 (1, 2)** → config의 `required_keys`를 기준으로 보완하라. `title`은 파일명에서, `type`은 폴더·내용에서 추론하고, 확신이 없으면 `status: draft`로 둔다. 볼트에 프런트매터 스키마 노트가 있으면(예: `00-meta/schemas/` 아래) 그 스키마를 우선 따르라.
- **Enum 위반 (3)** → config `enums`의 유효값 중 의미가 가장 가까운 값으로 교정하라 (예: `done` → `archive`, `wip` → `draft`). 애매하면 해당 노트의 맥락을 읽고 판단하라.
- **따옴표 없는 위키링크 (4)** → 해당 프런트매터 값을 이중 따옴표로 감싸라: `related: [[X]]` → `related: ["[[X]]"]`. 링크 대상을 바꾸지 마라.
- **로그 연산 태그 누락/미승인/형식오류 (9)** → config `log_note`의 해당 항목을 Edit로 고쳐라.
  - 태그 없음: 요약 맨 앞에 config `log_tags` 중 내용에 맞는 태그를 달아라 (판별 기준: 소스 소화=ingest · 리서치/질의=query · 무결성 점검=lint · 산출물 제작=build · 구조/규칙 변경=ops · 결정 확정=decision).
  - 미승인 태그: `log_tags`의 유효 태그로 치환하라.
  - 형식오류: `- YYYY-MM-DD HH:MM | 행위자 | [태그] 요약` 표준형으로 재배열하라. **내용(날짜·행위자·요약)은 보존하고 형식만 교정한다.**
  - `log_tag_epoch` 이전 항목은 검사 대상이 아니다 — 건드리지 마라.

## 4. 관리성 이슈 — 사용자 확인 후 처리 (섹션 4b, 5, 6, 6b, 6c, 7, 8)

발견 목록과 제안 조치를 먼저 표로 보여주고, **승인받은 항목만** 처리하라.

- **프런트매터 과대 (4b)** → 초과분(중복 related, 본문성 메타데이터)을 정리해 `frontmatter_max_lines` 이내로 줄이는 안을 제안하라.
- **데드 링크 (5)** → 두 갈래로 분류해 제안하라:
  - 오타/개명 흔적(기존 노트와 이름이 유사) → 링크 교정.
  - 개념상 존재해야 할 노트 → 프런트매터를 갖춘 스텁 노트 생성(`status: draft`).
- **고아 노드 (6)** → 관련 노트를 찾아 양방향 위키링크 주입, 또는 `index_note`에 등록을 제안하라.
- **준고아 (6b)** → 이 노트를 참조해야 할 상위/관련 노트에 링크 추가를 제안하라.
- **인덱스 미등록 (6c)** → config `index_note`의 적절한 섹션에 위키링크로 등록하라.
- **노화 문서 (7, `stale_days` 설정 시만)** → 승인된 것만 아카이브하되 순서를 지켜라: **① 프런트매터를 먼저 갱신**(`status: archive`, `ai_priority: archive`) → **② 셸 move로 deny zone(예: `20-knowledge/_archive/`)에 이동, 파일명은 절대 바꾸지 마라**(위키링크는 파일명 기반). 이동 후에는 deny 규칙 때문에 읽기·수정이 불가능하므로 갱신은 반드시 이동 전에 끝내라.
- **SSOT 사실 모순 (8, `ssot_facts` 설정 시만)** → 같은 식별자가 여러 값으로 표기된 결정론 누수다. **임의로 하나를 고르지 마라.** config `ssot_note`가 설정되어 있으면 그 노트에서 확정 값을 확인해 다른 노트를 통일하고, 확정 값이 없거나 `ssot_note`가 비어 있으면 사용자에게 어느 값이 옳은지 정합을 요청하라(출처 검증 없이 단정 금지). 정합 후에는 값을 SSOT 한 곳에만 두고 나머지 노트는 위키링크 참조로 유도한다.

## 5. 재검증 (fail-closed)

치유를 마쳤으면 1의 명령을 다시 실행해 **치명 이슈 0건(exit 0)** 을 확인하라. 여전히 치명이 남아 있으면 남은 항목과 이유를 보고하라 — 조용히 넘어가지 마라.

## 6. 기록과 보고

1. config `log_note`가 설정되어 있고 파일이 존재하면, 최상단에 수정 내역 1줄을 추가하라. 형식: `- YYYY-MM-DD HH:MM | Claude | [lint] 치명 N건 치유·관리성 M건 처리 (요약)`.
2. 수정 전/후 통계를 표로 보고하라: 섹션별 이슈 수(전→후), 즉시 치유 건수, 사용자 승인 처리 건수, 보류 건수.
